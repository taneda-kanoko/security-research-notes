# npm Squatting Run 2: SBOM を最小検知条件として検証（2026-06-04）

> 個人研究メモ。[Run 1](claude-code-npm-squatting-2026.md) で確認した「1ビット前提条件問題」を、SBOM 保有によって解消できるかを検証。

---

##### Background（背景）

Run 1（2026-05-16）では、Claude Code npm スクワッティング攻撃に対する3つの検知仮説（H1〜H3）を思考実験で検証し、次の核心的な脆弱性を特定した。

> **1ビット前提条件問題**  
> 「漏洩パッケージ名を把握しているか否か」という1ビットの情報が、検知率 100%（Case A）と 0%（Case B）を完全に分断する。  
> 技術的インフラの整備状況ではなく、**情報優位の有無**が構造的脆弱性の本体。

| 条件 | 勝者 | 検知 |
|---|---|---|
| Case A：漏洩情報あり | Blue（防御側） | H2 + H3 確定 |
| Case B：漏洩情報なし | Red（攻撃側） | 全シグナル不可視 |

Run 2 の問いは「Case B でも検知可能にする最小条件は何か」。
候補として SBOM（Software Bill of Materials）の事前整備を設定した。

---

##### Hypothesis（検証仮説）

**H4: SBOM 保有組織は Case B でも npm スクワッティングを検知できる**

SBOM には内部依存パッケージ名（`color-diff-napi` 等）が登録されている。  
これを定期的に npm registry と照合することで、H2 と同等の命名一致検知が  
「インシデント依存」ではなく「セキュリティポスチャ依存」として機能するはずである。

**初期予測: 65%**

Run 1 では H2 の初期予測が 30% だったが実際は 90%（35pp 過小評価）。  
その教訓を踏まえ控えめに設定したが、それでも過小評価になると予想した。

---

##### Method（実験条件）

**Run 2 の設定**

| 変数 | Run 1 | Run 2 |
|---|---|---|
| Blue の情報源 | 漏洩報道 + 漏洩ソース参照（Case A） | 漏洩報道のみ（Case B）+ **SBOM 保有** |
| SBOM 品質 | なし | high（全内部 -napi パッケージ記載済み） |
| npm 監視インフラ | なし | なし（H4 単独の効果を分離するため追加しない） |

**検証手順**

1. Blue が SBOM から内部パッケージ名リスト（5件）を取得
2. 各パッケージ名を npm registry API で照合

```
GET registry.npmjs.org/color-diff-napi   → pacifier136 / 2026-04-01 → ANOMALY
GET registry.npmjs.org/modifiers-napi    → pacifier136 / 2026-04-01 → ANOMALY
GET registry.npmjs.org/audio-capture-napi → pacifier136 / 2026-04-01 → ANOMALY
（以下 2 件同様）
```

3. 「SBOM に記載の内部名が第三者によって突然 npm に登録されている」を検知
4. H2・H3 の補強チェーンを起動（Run 1 と同一手順）

---

## Run 2 検証結果（2026-06-04）

| 仮説 | 内容 | Verdict | 初期予測 | 実際 | 予測精度 |
|---|---|---|---|---|---|
| **H4** | SBOM 照合で Case B 検知 | Confirmed | 65% | 90% | **25pp 過小評価** |
| **H2** | 命名規則照合（SBOM 起点） | Confirmed | — | 90% | — |
| **H3** | スタブ構成（H2 補完） | Confirmed | — | 60% | — |
| **H1** | アカウント行動（二次検知） | Partial | — | 40% | — |

**勝者**: Blue（防御側）

**予測精度の振り返り**:  
H4 の過小評価幅は 25pp。Run 1 の H2 では 60pp だったため改善している。  
ただし原因は同じ——「照合型検知の単純さ（API 1 コールで完結）」を再び過小評価した。  
SBOM という新しい情報源に切り替わっても、照合メカニズム自体の有効性は変わらない。

---

##### Key Finding（核心的発見）

**SBOM = Case B における最小検知条件として確定**

```
Case A（Run 1）: 漏洩発生 → 知識獲得 → 照合 → 検知      [反応的]
Case B + SBOM  : 平時整備 → スケジュール照合 → 検知      [予防的]
```

Case A と Case B + SBOM の「npm registry 照合」以降のステップは完全に同一。  
差分は**パッケージ名の入手経路のみ**——インシデント依存か、組織的準備依存か。

この転換が意味すること:
- 「漏洩を知った後に動く」から「常に監視している状態に移行する」
- 検知の前提条件が再現不可能な外部イベントから、計画・自動化可能な内部プロセスに変わる

---

##### Practical Implications（実務への示唆）

**SOC / CSIRT での自動化方針**

```
1. SBOM に内部パッケージ名を登録・維持（公開前の内部専用パッケージを含む）
2. 定期バッチ（例: 日次）で全エントリを npm registry API に照合
3. 「登録なし → 第三者が突然登録」のパターンをアラートに変換
4. 検知後は Run 1 の H2 + H3 補強チェーンで確度を上げてから対応
```

**インシデント前に検知できる条件**:
- SBOM が最新状態で維持されていること
- 照合バッチがスクワッティング公開後（今回は 2026-04-01 以降）に1回以上実行されること
- npm registry API への外部アクセスが可能であること

---

##### Limitations（限界）

1. **SBOM 品質依存**  
   SBOM が古い・不完全（内部限定パッケージが未登録）の場合、false negative が発生する。  
   実務では SBOM の更新頻度と内部パッケージの登録プロセスがボトルネックになる。

2. **H1（アカウント行動検知）は Partial のまま**  
   npm 監視インフラ（RSS / Webhook）なしでは、新規パッケージの一括公開パターンは  
   H4 → H2 発見後の二次確認にとどまる。スタンドアロン検知には別途インフラが必要。

3. **単一ケーススタディの限界**  
   今回の検証は Claude Code npm スクワッティング事例（2026-04-01）に基づく思考実験。  
   異なる攻撃者 OPSEC（例: パッケージ名を意図的に変形させる手法）では H4 が機能しない可能性がある。

4. **実環境未検証**  
   Proxmox 実環境での検証は 2028 年以降を予定。  
   本結果は思考実験のため、実際のログ・タイミング・誤検知率は未測定。
